## Part 1

1. _Чому аудіо-характеристики винесені в окремий об’єкт `audio_features`, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?_

   Винесення в `audio_features` застосовується для логічного групування даних (Data Cohesion).

   **Вигідно:**
   - Коли клієнту або фронтенду потрібні всі технічні метрики треку одразу, їх можна отримати одним запитом не перелічуючи десяток окремих полів.
   - Документ стає більш читабельним, відділяючи базові метадані (назва, альбом, тривалість).

   \*Створює проблеми:\*\*
   - Звернення до вкладених полів вимагає використання dot-notation, що синтаксично трохи складніше.
   - Коли потрібно часто оновлювати лише одне поле всередині об'єкта, розробник має бути обережним і обов'язково використовувати оператор `$set`, інакше можна випадково перезаписати весь об'єкт `audio_features`.

2. _Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?_
   - Зберігання у вигляді масиву дозволяє MongoDB створити Multikey Index, де кожен елемент індексується як окреме значення.

   - **Чому не рядок:** Пошук артиста у рядку типу `"Artist A; Artist B"` вимагає використання RegEx для пошуку по підрядку. Це вкрай повільна операція, яка ігнорує звичайні індекси і може давати хибні результати.

   **Які запити стають простішими:**
   - Точний пошук: `{ artists: "Name" }`миттєво знайде всі треки виконавця, незалежно від того,скільки їх там.
   - Перетини: Використання оператора $all дозволяє легко знайти треки, де працювали конкретні 2-3 артисти разом.
   - Агрегація: Масив дозволяє використовувати оператор $unwind, щоб "розгорнути" трек і зробити аналітику по кожному артисту окремо.

3. _Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?_

   Обидва оператори є фінальними стадіями Aggregation Pipeline, які записують результати у колекцію.
   - `$out`: Повністю перезаписує цільову колекцію. Якщо колекція вже існувала, вона видаляється, і на її місце записуються нові дані.

     **Використовувати:** міграція даних з нуля, створення повних резервних копій.

   - `$merge`: Оновлює існуючу колекцію. Дозволяє гнучко налаштувати логіку: що робити, якщо документ з таким `_id` вже існує.

     **Використовувати:** для постійних оновлень, не видаляючи та не перераховуючи мільйони існуючих треків.

## Part 2

1. _Для чого використовується інструкція `$unwind`?_

   Оператор `$unwind` використовується для деконструкції масиву. Він створює окрему копію документа. Без цього неможливо провести агрегацію по окремих елементах (порахувати середню популярність для кожного артиста окремо, якщо в треку їх 2-3).

2. _Чим `$stdDevPop` відрізняється від `$stdDevSamp`?_
   Обидва оператори рахують середньоквадратичне відхилення, але за різними формулами:
   - `$stdDevPop`: використовується, коли дані є повною генеральною сукупністю.
   - `$stdDevSamp`: використовується, коли дані є лише вибіркою.

## Part 4

### Task 1

1. _Що змінилося в плані виконання?_
   - **Швидкість:** Час виконання запиту скоротився з 72 мс до 2 мс.
     - Before: `"executionTimeMillis": 72`
     - After: `"executionTimeMillis": 2`

   - **Ефективність:** Кількість переглянутих документів зменшилася зі 113,999 (вся база) до 354 (тільки ті, що відповідають умові).
   * Before: `"totalDocsExamined": 113999`
   * After: `"totalDocsExamined": 354`
   - **Алгоритм:** Система перейшла від повного сканування колекції (COLLSCAN) до сканування індексу (IXSCAN). Стадія SORT зникла, оскільки індекс вже зберігає дані у відсортованому за популярністю вигляді.

2. _Як зрозуміти, що індекс використовується?_

- Поле `stage` має значення IXSCAN, що вказує на сканування індексу.
- Поле `totalKeysExamined = 412` => що означає база даних використовувала ключі індексу для пошуку.
- Поле `indexName` містить назву створеного ніндексу: `track_genre_1_popularity_-1_audio_features.danceability_1`.

```
{ 
    "executionSuccess": true,
    "executionTimeMillis": 2,
    "totalKeysExamined": 412,
    "totalDocsExamined": 354,
    "executionStages": {
        ... ,
        "inputStage": {
            "stage": "IXSCAN",
            ...,
            "indexName": "track_genre_1_popularity_-1_audio_features.danceability_1",
            ...
            }
    }
}
```

### Task 3

Цей запит не є покривним, оскільки він не містить проєкції, яка б обмежувала вивід лише індексованими полями. MongoDB змушена звертатися до самої колекції (стадія FETCH), щоб отримати інші поля документа, а також поле _id, яких немає у структурі створеного індексу

***Як зробити його покривним?***

Потрібно додати проєкцію, яка обмежить результат лише полями, що є в індексі, та обов'язково виключити _id:

```
db.tracks.find({ track_genre: "pop", popularity: { $gte: 70 } }, 
               { track_genre: 1, popularity: 1, _id: 0 });
```
